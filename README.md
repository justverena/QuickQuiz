# QuickQuiz

## Описание проекта

**QuickQuiz** — это система для создания, прохождения и анализа онлайн-викторин.  
Проект построен по микросервисной архитектуре, где каждый сервис отвечает за свою часть логики — квизы, пользователи и логи.  

---

## Архитектура проекта

Проект состоит из трех микросервисов:

| Сервис | Назначение |
|--------|-------------|
| **Quiz Service** | Управляет вопросами и квизами, обрабатывает ответы пользователей |
| **Auth Service** | Отвечает за регистрацию, вход и управление пользователями |
| **Log Service** | Хранит и анализирует логи действий пользователей и системные события |

## Ответы на вопросы:

### Как выбор структуры базы данных (SQL или NoSQL) влияет на дизайн CRUD API?

Если база SQL — нужно чётко задавать таблицы и связи, API будет работать со связанными данными.
Если NoSQL — данные хранятся свободнее, API проще, но проверки и связи нужно делать вручную.

### Какие проблемы могут возникнуть при массовых обновлениях данных через API?

При массовых обновлениях система начнёт работать медленно или выдавать ошибки; могут возникнуть конфликты, если несколько запросов меняют одни и те же данные, и часть изменений потеряется. Также возможны ошибки сети, из-за которых данные обновятся не полностью.

### Почему важно использовать правильные HTTP-методы (GET, POST, PUT, DELETE), а не только POST?

Потому что каждый метод имеет своё назначение: GET — получать, POST — добавлять, PUT — обновлять, DELETE — удалять.
Если всё делать через POST, система станет непонятной, сложной для поддержки и может работать неправильно.

### Какие уязвимости могут возникнуть при хранении JWT на клиентской стороне?

Основаная уязвимость - это риск похищения токена из localStorage, sessionStorare или из cookie с целью CSRF атаки. Как правило рекомендуется хранить токены в cookies с помощью HttpOnly, но если приходится хранить их в localStorage, то время жизни access токена уменьшают, а в cookies хранят только refresh токен. Также возможно использование Content Security Policy на фронте.

### В каких случаях стоит ограничивать время жизни JWT, и какие проблемы это создает для UX?

Время жизни токенов уменьшают когда токены хранятся в localStorage чтобы уменьшить риск его использования злоумышнниками даже если он попадет к ним в руки. Также его уменьшают для запросов с чувствительными данными, как например банковские операции или личные сообщения. Для опыта пользователей это может создать небольшие неудобства, потому что это означает более частые обновления токенов, что может приводить к повторному вводу данных если пользователь промедлил.

### Как логирование помогает в расследовании инцидентов безопасности?

Логирование фиксирует все действия в системе — кто, когда и что делал, это помогает понять как произошёл инцидент, и найти его причину.
Еще по логам можно быстро заметить подозрительное поведение, например большое количество неудачных попыток входа или необычные запросы. Из-за этого можно быстрее реагировать на угрозы, найти уязвимости и в дальнейшем улучшить безопасность системы.
